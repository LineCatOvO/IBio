## 一、FIDO2 协议概述

### 1.1 什么是 FIDO2

FIDO2 是 FIDO Alliance（Fast IDentity Online Alliance）制定的新一代身份认证标准，旨在消除密码认证的弱点，提供更安全、更便捷的用户认证方式。

FIDO2 由两个核心协议组成：

| 协议 | 全称 | 作用 | 标准组织 |
|------|------|------|----------|
| **WebAuthn** | Web Authentication | 浏览器与认证器之间的标准 API | W3C |
| **CTAP** | Client to Authenticator Protocol | 客户端（浏览器/平台）与认证器之间的通信协议 | FIDO Alliance |

### 1.2 FIDO2 认证流程

FIDO2 认证流程分为两个阶段：

**注册阶段（Registration）**：
```
1. 用户访问网站，请求注册
2. 服务器生成挑战（challenge）和用户信息
3. 浏览器调用 WebAuthn API，请求认证器生成凭证
4. 认证器生成密钥对，存储私钥，返回公钥和凭证 ID
5. 浏览器将凭证信息返回给服务器
6. 服务器存储凭证 ID 和公钥，关联用户账户
```

**认证阶段（Authentication）**：
```
1. 用户访问网站，请求登录
2. 服务器生成挑战，发送凭证 ID 列表
3. 浏览器调用 WebAuthn API，请求认证器签名
4. 认证器验证用户（指纹/PIN），使用私钥签名挑战
5. 浏览器将签名结果返回给服务器
6. 服务器验证签名，完成认证
```

### 1.3 FIDO2 核心优势

| 优势 | 说明 |
|------|------|
| **无密码** | 消除密码泄露、弱密码、密码 reuse 等风险 |
| **抗钓鱼** | 认证器绑定域名，无法跨域使用凭证 |
| **隐私保护** | 认证器不存储用户信息，服务器不存储私钥 |
| **本地验证** | 用户验证（指纹/PIN）在认证器本地完成 |
| **强加密** | 使用非对称加密（公钥加密、私钥签名） |

## 二、CTAP 协议详解

### 2.1 CTAP 协议版本对比

| 版本 | 协议名称 | 发布时间 | 主要特性 | 认证器类型 |
|------|----------|----------|----------|------------|
| **CTAP1/U2F** | Universal 2nd Factor | 2014 | 仅支持第二因素认证、仅 ES256 算法 | U2F 认证器 |
| **CTAP2/FIDO2** | Client to Authenticator Protocol v2 | 2018 | 支持第一因素认证、多种算法、用户验证 | FIDO2 认证器 |
| **CTAP2.1** | CTAP v2.1 | 2021 | 增强安全特性、支持 PIN/生物识别变化检测 | FIDO2 认证器 |

### 2.2 CTAP1/U2F 协议核心机制

**工作模式**：仅作为第二因素认证，配合密码使用

**核心流程**：
```
1. 用户输入密码完成第一因素认证
2. 用户触摸/插入 U2F 认证器
3. 认证器使用私钥签名挑战
4. 服务器验证签名，完成第二因素认证
```

**局限性**：
- 仅支持 ES256 算法（ECDSA with SHA-256）
- 不支持用户验证（无 PIN/生物识别）
- 不支持凭证管理（无法删除/枚举凭证）
- 仅支持 USB 传输

### 2.3 CTAP2/FIDO2 协议核心机制

**工作模式**：支持第一因素认证（单因素）和第二因素认证

**新增特性**：

| 特性 | 说明 |
|------|------|
| **用户验证** | 支持 PIN、指纹、面部识别等 |
| **凭证管理** | 支持枚举、删除凭证 |
| **多种算法** | 支持 ES256、RS256、EdDSA |
| **resident credential** | 凭证可存储在认证器内，无需服务器存储 |
| **多传输协议** | USB、NFC、BLE、内部（平台认证器） |

### 2.4 CTAP 消息结构

CTAP 使用 CBOR（Concise Binary Object Representation）编码消息：

**消息格式**：
```
[Command Byte] [CBOR Encoded Parameters]
```

**常用命令**：

| 命令 | 命令字节 | 说明 |
|------|----------|------|
| `MakeCredential` | 0x01 | 创建新凭证 |
| `GetAssertion` | 0x02 | 获取认证签名 |
| `GetInfo` | 0x04 | 获取认证器信息 |
| `ClientPIN` | 0x06 | PIN 管理 |
| `Reset` | 0x07 | 重置认证器 |
| `GetNextAssertion` | 0x08 | 获取下一个认证（多凭证场景） |
| `GetCredentialCount` | 0x11 | 获取凭证计数 |

### 2.5 CTAP2.1 新增特性

| 特性 | 说明 |
|------|------|
| **PIN/生物识别变化检测** | 检测 PIN 变化或生物识别模板变化 |
| **enterprise attestation** | 企业级认证器标识 |
| **credential protection** | 凭证保护级别（UVRequired、UVOptional） |
| **always UV** | 强制用户验证模式 |

## 三、核心算法要求

### 3.1 支持的算法

FIDO2 标准支持的算法在 COSE（CBOR Object Signing and Encryption）中定义：

| 算法标识 | 算法名称 | COSE Value | 说明 | 必须支持 |
|----------|----------|------------|------|----------|
| **ES256** | ECDSA with SHA-256 | -7 | P-256 曲线，推荐算法 | 必须 |
| **RS256** | RSASSA-PKCS1-v1_5 with SHA-256 | -257 | RSA 2048+ 位，可选 | 推荐 |
| **EdDSA** | EdDSA with SHA-512 | -8 | Ed25519 曲线，可选 | 推荐 |
| **ES384** | ECDSA with SHA-384 | -35 | P-384 曲线，可选 | 不要求 |
| **ES512** | ECDSA with SHA-512 | -36 | P-521 曲线，可选 | 不要求 |

### 3.2 ES256 算法详解

**算法名称**：ECDSA with SHA-256
**曲线**：NIST P-256（secp256r1）
**公钥格式**：COSE Key 格式， uncompressed 格式（65 字节）

**签名过程**：
```
1. 生成随机数 k
2. 计算 R = k * G（曲线基点）
3. 计算 r = R.x mod n（曲线阶数）
4. 计算 s = k^-1 * (z + r * d) mod n（z 为消息哈希，d 为私钥）
5. 签名结果 (r, s)
```

**验证过程**：
```
1. 验证 r, s 在有效范围内
2. 计算 w = s^-1 mod n
3. 计算 u1 = z * w mod n
4. 计算 u2 = r * w mod n
5. 计算 R = u1 * G + u2 * Q（Q 为公钥）
6. 验证 r ≡ R.x mod n
```

### 3.3 ECDH 密钥协商

用于 PIN 加密和凭证保护：

**算法**：ECDH with P-256
**用途**：
- PIN 建立加密通道
- 凭证加密存储
- 认证器与客户端密钥协商

**流程**：
```
1. 客户端生成临时密钥对 (pinUvAuthTokenPubKey)
2. 认证器生成临时密钥对
3. 双方交换公钥
4. 计算共享密钥 K = ECDH(privateKey, peerPublicKey)
5. 使用 K 加密敏感数据
```

### 3.4 SHA-256 哈希

**用途**：
- 挑战哈希
- 算法参数哈希（rpIdHash）
- 签名消息哈希
- PIN 哈希（左 16 字节）

**输入**：任意长度数据
**输出**：256 位（32 字节）哈希值

## 四、安全要求

### 4.1 密钥存储要求

| 要求 | 级别 | 说明 |
|------|------|------|
| **私钥不可导出** | 必须 | 私钥必须存储在认证器内，不可导出到外部 |
| **私钥不可读取** | 必须 | 私钥不可被读取，只能用于签名 |
| **密钥隔离** | 必须 | 不同凭证的密钥必须隔离存储 |
| **密钥加密存储** | 推荐 | 私钥应加密存储，防止物理攻击 |
| **防侧信道攻击** | 推荐 | 防止功耗、时间等侧信道攻击 |

### 4.2 防篡改要求

| 要求 | 级别 | 说明 |
|------|------|------|
| **固件防篡改** | 必须 | 固件不可被修改，防止恶意注入 |
| **固件签名验证** | 推荐 | 固件更新必须签名验证 |
| **硬件防篡改检测** | 推荐 | 检测物理篡改，触发密钥销毁 |
| **安全启动** | 推荐 | 启动过程验证固件完整性 |

### 4.3 用户验证要求

| 验证方式 | FIDO2 支持 | 安全级别 | 说明 |
|----------|------------|----------|------|
| **PIN** | 支持 | 中 | 4-8 位数字，本地验证 |
| **指纹** | 支持 | 高 | 生物识别，本地验证 |
| **面部识别** | 支持 | 高 | 生物识别，平台认证器常见 |
| **无验证** | 支持（第二因素） | 低 | 仅触摸确认 |

**用户验证标志（UV）**：
- `UV=1`：用户验证成功（PIN/生物识别）
- `UV=0`：用户验证未执行
- `UP=1`：用户在场确认（触摸）

### 4.4 认证器级别

FIDO Alliance 定义了认证器安全级别：

| 纺别 | 名称 | 说明 | 认证要求 |
|------|------|------|----------|
| **L1** | Basic | 基础安全级别 | 功能测试通过 |
| **L2** | Basic+ | 增强安全级别 | 功能测试 + 安全测试通过 |
| **L3** | Advanced | 高级安全级别 | 功能测试 + 安全测试 + 硬件安全测试 |
| **L3+** | Advanced+ | 最高安全级别 | L3 + 防物理攻击 |

### 4.5 attestation（认证器认证）

**目的**：证明认证器是真实的 FIDO 认证器

**类型**：

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| **None** | 不提供认证器认证 | 普通用户场景 |
| **Self** | 认证器自我签名认证 | 低安全要求场景 |
| **Basic** | 基础批量认证 | 企业批量采购 |
| **AttCA** | 批量认证 CA | 企业场景 |
| **ECDAA** | 直接匿名认证 | 高隐私要求场景 |

## 五、生物识别集成要求

### 5.1 指纹集成要求

| 要求 | 说明 |
|------|------|
| **本地处理** | 指纹采集、比对必须在认证器本地完成 |
| **模板隔离** | 指纹模板不可导出，存储在安全区域 |
| **误识率（FAR）** | 建议 < 0.001% (1/100,000) |
| **拒识率（FRR）** | 建议 < 3% |
| **模板数量** | 建议支持 5+ 个指纹模板 |
| **匹配时间** | 建议 < 1 秒 |

### 5.2 指纹验证流程

```
1. 用户触摸认证器
2. 指纹传感器采集指纹图像
3. 提取指纹特征（minutiae）
4. 与存储模板比对
5. 匹配成功 → 设置 UV=1，继续认证流程
6. 匹配失败 → 返回错误，拒绝认证
```

### 5.3 指纹模板管理

| 操作 | 说明 |
|------|------|
| **注册** | 采集多次指纹，生成平均模板 |
| **删除** | 删除指定指纹模板 |
| **枚举** | 返回已注册指纹模板列表（仅友好名） |
| **更新** | 更新指纹模板友好名 |

### 5.4 指纹安全要求

| 要求 | 说明 |
|------|------|
| **模板加密** | 指纹模板应加密存储 |
| **模板不可导出** | 指纹模板不可导出到外部 |
| **防假指纹** | 检测假指纹攻击（硅胶、照片等） |
| **防重放** | 每次认证生成随机挑战，防止重放 |

## 六、通信协议要求

### 6.1 USB 传输要求

| 要求 | 说明 |
|------|------|
| **接口类型** | USB HID（Human Interface Device） |
| **协议** | CTAP over HID |
| **数据包大小** | 64 字节 |
| **端点** | HID Interrupt IN/OUT |
| **超时** | 响应超时建议 30 秒 |

### 6.2 NFC 传输要求

| 要求 | 说明 |
|------|------|
| **协议** | CTAP over NFC（ISO 14443 Type A/B） |
| **APDU 格式** | ISO 7816-4 APDU |
| **数据交换** | 通过 APDU 命令/响应 |
| **超时** | 建议响应超时 5 秒 |

### 6.3 BLE 传输要求

| 要求 | 说明 |
|------|------|
| **协议** | CTAP over BLE（Bluetooth Low Energy） |
| **GATT 服务** | FIDO Service UUID: 0xFFFD |
| **特征值** | Control Point, Status, Service Revision |
| **连接参数** | 建议 Connection Interval 15-30 ms |

### 6.4 多传输认证器

支持多种传输协议的认证器（Cross-platform Authenticator）：

| 组合 | 说明 |
|------|------|
| **USB + NFC** | 常见组合，YubiKey 5 NFC |
| **USB + NFC + BLE** | 全面支持，YubiKey 5Ci |
| **USB only** | 基础认证器 |

## 十、结论

IBio 项目需要实现 FIDO2 标准认证器，核心要求如下：

1. **必须实现**：MakeCredential、GetAssertion、用户验证（PIN/指纹）、ES256 签名、USB 传输
2. **推荐实现**：resident credential、凭证管理、NFC 传输、多算法支持
3. **安全要求**：私钥不可导出、防篡改、用户验证本地完成
4. **生物识别**：指纹本地比对、模板加密存储、误识率 < 0.001%
5. **性能要求**：认证时间 < 2 秒、功耗 < 100 mA、成本 < $30
