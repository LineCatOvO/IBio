# ADR-001: 选择 ESP32-S3 + FreeRTOS 技术栈

**状态**：已接受（Accepted）

**决策日期**：2026-04-06

**决策者**：IBio 项目团队

---

## 一、标题

选择 ESP32-S3 + FreeRTOS（ESP-IDF）+ C + PlatformIO 技术栈实现 IBio FIDO2 安全密钥

---

## 二、背景

### 2.1 项目需求

IBio 项目是一个 DIY FIDO2 安全密钥，需要实现：

1. **FIDO2 标准**：符合 WebAuthn 和 CTAP2.0/2.1 协议
2. **生物识别**：集成指纹验证
3. **双模通信**：USB + BLE
4. **低成本**：目标成本 < $31
5. **开源**：便于社区复用

### 2.2 技术选型约束

| 约束条件 | 说明 |
|----------|------|
| **硬件约束** | 需要 USB HID 和 BLE 支持 |
| **安全约束** | 需要安全芯片进行密钥存储和签名 |
| **成本约束** | 总硬件成本不超过 $31 |
| **开发约束** | 需要成熟的开发工具链和社区支持 |
| **时间约束** | DIY 项目，开发周期不宜过长 |

### 2.3 决策触发

需要选择主控芯片、固件框架、开发语言和开发工具链，以便后续开发工作能够顺利开展。

---

## 三、决策

### 3.1 最终决策

选择以下技术栈：

| 类别 | 选择 | 具体方案 |
|------|------|----------|
| **主控芯片** | ESP32-S3 | ESP32-S3-WROOM-1 模块 |
| **指纹传感器** | FPM383C | 电容式指纹模块 |
| **安全芯片** | ATECC608A | Microchip 安全芯片 |
| **固件框架** | FreeRTOS | ESP-IDF 内置 FreeRTOS |
| **开发语言** | C | ESP-IDF 标准 C 语言 |
| **开发工具链** | ESP-IDF | 官方 ESP-IDF 工具链 |
| **开发环境** | VS Code + PlatformIO | 跨平台 IDE |
| **测试工具** | Unity + FIDO2 Test Tool | 单元测试 + 认证测试 |

### 3.2 决策理由

选择该技术栈的核心理由：

1. **USB + BLE 内置**：ESP32-S3 内置 USB OTG 和 BLE 5.0，无需外接 USB/BLE 芯片，减少硬件复杂度和成本
2. **成熟开发框架**：ESP-IDF 是官方框架，文档完善，示例丰富，社区活跃
3. **开源项目参考**：SoloKeys 项目使用类似架构，有成熟参考
4. **成本适中**：ESP32-S3 成本 $5-8，总成本 $21-31，符合目标
5. **开发便捷**：PlatformIO 提供统一的开发环境，简化构建和调试

## 四、考虑的选项

### 4.1 主控芯片选项对比

| 选项 | 优点 | 缺点 | 成本 |
|------|------|------|------|
| **ESP32-S3** | USB + BLE 内置、价格适中、社区活跃 | 功耗较高、无硬件安全特性 | $5-8 |
| **nRF52840** | BLE + USB 内置、低功耗、高安全性 | 价格较高、开发复杂 | $5-7 |
| **RP2040** | 价格极低、USB 内置、社区活跃 | 无 BLE、无硬件安全 | $1-2 |
| **STM32L4** | 超低功耗、硬件安全、成熟生态 | 价格高、无 BLE、开发复杂 | $5-8 |

**对比结论**：
- ESP32-S3 在 USB + BLE 双模支持方面最优
- nRF52840 在低功耗和安全性方面最优，但价格较高
- RP2040 成本最低，但功能受限
- STM32L4 适合单一场景，不适合双模

**选择 ESP32-S3 理由**：
1. USB + BLE 双模内置，减少外接芯片
2. 成本适中，符合目标
3. 开发便捷，ESP-IDF 完善
4. 社区活跃，资料丰富

### 4.2 指纹传感器选项对比

| 选项 | 优点 | 缺点 | 成本 |
|------|------|------|------|
| **FPM383C** | 价格适中、内置算法、小体积 | 模板容量有限（200） | $10-15 |
| **GT521F32** | 性能高、模板容量大（1000+） | 价格较高、体积较大 | $15-20 |
| **R306** | 价格低、光学式稳定 | 功耗高、体积大 | $10-15 |
| **AS608** | 国产、价格低、中文资料 | 社区资源少 | $8-12 |

**对比结论**：
- FPM383C 在成本、体积、性能平衡最优
- GT521F32 性能最优，但价格高
- R306 成本低但体积大
- AS608 国产方案，资料少

**选择 FPM383C 理由**：
1. 价格适中，符合成本目标
2. 内置指纹算法，减少开发工作量
3. UART 接口简单，易于集成
4. 小体积适合便携设备

### 4.3 安全芯片选项对比

| 选项 | 优点 | 缺点 | 成本 |
|------|------|------|------|
| **ATECC608A** | 价格极低、开源项目验证、功能满足 | 安全等级较低（EAL4+） | $1-2 |
| **OPTIGA Trust M** | 高安全等级（EAL6+）、功能丰富 | 价格较高 | $2-4 |
| **SE050** | 高安全等级、NXP 生态 | 价格高、开发复杂 | $3-5 |
| **STSAFE-A100** | STM32 集成好、适中安全 | 功能较少 | $2-3 |

**对比结论**：
- ATECC608A 成本最优，开源项目验证
- OPTIGA Trust M 安全最优，但价格高
- SE050 功能丰富，但复杂
- STSAFE-A100 适合 STM32 项目

**选择 ATECC608A 理由**：
1. 价格极低，显著降低成本
2. SoloKeys、OpenSK 项目验证可用
3. 满足 ES256 签名要求
4. CryptoAuthLib 库完善

### 4.4 固件框架选项对比

| 选项 | 优点 | 缺点 |
|------|------|------|
| **FreeRTOS（ESP-IDF）** | ESP32 官方、成熟稳定、集成 BLE | 无内置 CTAP 实现 |
| **Zephyr RTOS** | 内置 CTAP 参考、模块化、OpenSK 参考 | 学习曲线高、ESP32 支持有限 |
| **Arduino** | 极简单、社区活跃 | 非实时、不适合生产 |
| **裸机** | 代码最小、完全控制 | 开发复杂、无多任务 |

**对比结论**：
- FreeRTOS 成熟稳定，ESP-IDF 集成完善
- Zephyr 有 CTAP 参考，但 ESP32 支持有限
- Arduino 适合原型，不适合生产
- 裸机开发复杂

**选择 FreeRTOS（ESP-IDF）理由**：
1. ESP32 官方框架，集成完善
2. FreeRTOS 内核成熟稳定
3. BLE/USB 驱动内置
4. SoloKeys 项目参考架构

### 4.5 开发语言选项对比

| 选项 | 优点 | 缺点 |
|------|------|------|
| **C** | ESP-IDF 标准、库丰富、性能高 | 安全性较低、手动管理内存 |
| **Rust** | 内存安全、现代开发 | 嵌入式生态增长中、学习曲线高 |
| **MicroPython** | 极简单、交互式 | 性能低、不适合实时 |
| **C++** | 面向对象、性能高 | 复杂、嵌入式应用少 |

**对比结论**：
- C 是 ESP-IDF 标准语言，成熟稳定
- Rust 安全性最优，但生态增长中
- MicroPython 不适合实时系统
- C++ 嵌入式应用少

**选择 C 理由**：
1. ESP-IDF 标准语言
2. 库生态极丰富
3. 学习资料多
4. SoloKeys/OpenSK 参考（部分）

### 4.6 开发工具链选项对比

| 选项 | 优点 | 缺点 |
|------|------|------|
| **PlatformIO** | 跨平台、库管理、VS Code 集成 | 依赖网络 |
| **ESP-IDF CLI** | 官方工具、功能完整 | 命令行复杂 |
| **GCC + Make** | 标准、灵活 | 配置复杂 |
| **Arduino IDE** | 极简单 | 功能受限 |

**对比结论**：
- PlatformIO 跨平台最优，VS Code 集成
- ESP-IDF CLI 官方工具，功能完整
- GCC + Make 标准但复杂
- Arduino IDE 简单但受限

**选择 PlatformIO 理由**：
1. 跨平台支持（Windows/Linux/macOS）
2. VS Code 集成，调试便利
3. 库管理器，依赖管理简单
4. 支持 ESP-IDF 框架

## 五、决策结果

### 5.1 最终技术栈

| 类别 | 选择 | 版本/型号 |
|------|------|-----------|
| **主控芯片** | ESP32-S3 | ESP32-S3-WROOM-1-N8R8 |
| **指纹传感器** | FPM383C | FPM383C 模块 |

| **开发语言** | C | C11 标准 |
| **开发工具链** | PlatformIO | Core 6.0+ |
| **开发环境** | VS Code | + PlatformIO 扩展 |
| **测试工具** | Unity + FIDO2 Test Tool | Unity 2.5+ |

### 5.2 技术栈验证

| 验证项 | 结果 |

| **ATECC608A ESP32 集成** | 确认可行，CryptoAuthLib 支持 ESP32 |

## 六、后果

### 6.1 正面影响

| 影响 | 说明 |
|------|------|
| **开发便捷** | ESP-IDF 和 PlatformIO 提供完善的开发工具链 |
| **成本适中** | 总成本 $21-31，符合项目目标 |
| **双模支持** | USB + BLE 内置，无需外接芯片 |
| **开源参考** | SoloKeys 项目提供成熟架构参考 |
| **社区活跃** | ESP32 社区活跃，资料丰富 |

### 6.2 负面影响

| 影响 | 说明 | 缓解措施 |
|------|------|----------|
| **功耗较高** | ESP32-S3 功耗比 nRF52 高 | 优化睡眠模式，减少唤醒频率 |
| **无硬件安全特性** | ESP32-S3 无硬件安全模块 | 使用 ATECC608A 外接安全芯片 |
| **C 语言安全性** | C 语言内存安全性较低 | 代码规范、静态分析、单元测试 |
| **无内置 CTAP** | ESP-IDF 无 CTAP 实现 | 参考 SoloKeys/OpenSK 实现 |
| **学习曲线** | ESP-IDF 有一定学习曲线 | 官方文档完善，示例丰富 |

### 6.3 技术风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| **USB HID 兼容性** | 低 | 中 | 使用 ESP-IDF USB HID 示例验证 |
| **BLE GATT 稳定性** | 中 | 中 | 测试 BLE 连接稳定性 |
| **ATECC608A 集成复杂度** | 中 | 中 | 参考 CryptoAuthLib 示例 |
| **FPM383C SDK 兼容性** | 低 | 低 | UART 协议标准化 |
| **固件体积过大** | 中 | 中 | 优化编译选项，裁剪功能 |

### 6.4 后续决策依赖

本决策影响以下后续决策：

1. **LLD 详细设计**：需要基于 ESP32-S3 和 ATECC608A 设计详细接口
2. **PCB 设计**：需要基于 ESP32-S3-WROOM-1 设计 PCB 布局
3. **固件架构**：需要基于 FreeRTOS 设计固件模块架构
4. **测试策略**：需要基于 Unity 和 FIDO2 Test Tool 设计测试方案

## 七、替代方案分析

### 7.1 替代方案一：nRF52840 + Zephyr + Rust

**方案描述**：
- 主控：nRF52840
- 固件框架：Zephyr RTOS
- 开发语言：Rust
- 参考：OpenSK 项目

**对比分析**：

| 项目 | nRF52840 + Zephyr | ESP32-S3 + FreeRTOS |
|------|-------------------|---------------------|
| **USB + BLE 内置** | 支持 | 支持 |
| **功耗** | 低功耗（优） | 较高功耗 |
| **安全等级** | 较高 | 依赖外接芯片 |
| **开发复杂度** | 高 | 中 |
| **成本** | $5-7 | $5-8 |
| **开源参考** | OpenSK | SoloKeys |
| **社区支持** | 中 | 高 |
| **学习曲线** | 高（Rust + Zephyr） | 中 |

**未选择原因**：
1. Rust 嵌入式生态仍在增长中
2. Zephyr 学习曲线较高
3. nRF52840 价格略高
4. ESP32 社区更活跃

### 7.2 替代方案二：RP2040 + 裸机 + C

**方案描述**：
- 主控：RP2040
- 固件框架：裸机
- 开发语言：C
- 成本：极低

**对比分析**：

| 项目 | RP2040 + 裸机 | ESP32-S3 + FreeRTOS |
|------|----------------|---------------------|
| **成本** | 极低（$1-2） | 适中（$5-8） |
| **USB 内置** | 支持 | 支持 |
| **BLE 内置** | 无（需外接） | 支持 |
| **开发复杂度** | 高（裸机） | 中 |
| **社区支持** | 高 | 高 |
| **硬件安全** | 无（需外接） | 需外接 |

**未选择原因**：
1. 无 BLE 内置，需外接 BLE 模块
2. 裸机开发复杂度高
3. 需外接更多芯片（BLE + 安全）
4. 总成本优势不明显

### 7.3 替代方案三：STM32L4 + FreeRTOS + C

**方案描述**：
- 主控：STM32L4
- 固件框架：FreeRTOS
- 开发语言：C
- 特点：低功耗

**对比分析**：

| 项目 | STM32L4 + FreeRTOS | ESP32-S3 + FreeRTOS |
|------|---------------------|---------------------|
| **功耗** | 超低功耗（优） | 较高功耗 |
| **USB 内置** | 支持（部分型号） | 支持 |
| **BLE 内置** | 无（需外接） | 支持 |
| **硬件安全** | 内置（部分型号） | 需外接 |
| **成本** | 较高（$5-8） | 适中（$5-8） |
| **开发复杂度** | 中 | 中 |
| **社区支持** | 中 | 高 |

**未选择原因**：
1. 无 BLE 内置，需外接 BLE 模块
2. STM32L4 价格较高
3. 需外接 BLE 芯片，增加成本
4. ESP32 社区更活跃
