# ADR-004: 选择 ATECC608A 作为安全芯片

**状态**：已接受（Accepted）

**决策日期**：2026-04-06

**决策者**：IBio 项目团队

---

## 一、标题

选择 Microchip ATECC608A-MAHDA-T 作为 IBio FIDO2 安全密钥的安全芯片

---

## 二、背景

### 2.1 项目需求

IBio 项目作为 FIDO2 安全密钥，需要满足以下安全需求：

| 需求类型 | 具体需求 | 来源 |
|----------|----------|------|
| **密钥存储** | 私钥不可导出、密钥隔离存储 | FIDO2 标准 |
| **签名功能** | 支持 ES256（ECDSA with SHA-256）签名 | CTAP2 协议 |
| **密钥协商** | 支持 ECDH for PIN 加密通道 | CTAP2 ClientPIN |
| **哈希功能** | 支持 SHA-256 哈希 | FIDO2 标准 |
| **密钥生成** | 硬件生成密钥对，私钥永不离开芯片 | FIDO2 安全要求 |
| **防篡改** | 防止物理攻击、侧信道攻击 | FIDO2 安全要求 |

### 2.2 技术约束

| 约束条件 | 说明 |
|----------|------|
| **主控芯片兼容性** | 需与 ESP32-S3 I2C 接口兼容 |
| **开发库支持** | 需有成熟的开源开发库 |
| **开源项目验证** | 优先选择已被开源项目验证的芯片 |
| **社区支持** | 需有活跃的社区和技术支持 |

### 2.3 成本约束

| 约束 | 目标值 | 说明 |
|------|--------|------|
| **安全芯片单价** | < $5 | 符合总成本 $20-31 目标 |
| **总硬件成本** | < $31 | 项目整体成本目标 |
| **推荐成本** | < $2 | 最大化成本效益 |

### 2.4 FIDO2 安全要求

根据 FIDO2 标准，认证器需满足以下安全要求：

| 要求 | 级别 | 说明 |
|------|------|------|
| **私钥不可导出** | 必须 | 私钥必须存储在认证器内，不可导出到外部 |
| **私钥不可读取** | 必须 | 私钥不可被读取，只能用于签名 |
| **密钥隔离** | 必须 | 不同凭证的密钥必须隔离存储 |
| **密钥加密存储** | 推荐 | 私钥应加密存储，防止物理攻击 |
| **防侧信道攻击** | 推荐 | 防止功耗、时间等侧信道攻击 |
| **防篡改检测** | 推荐 | 检测物理篡改，触发密钥销毁 |

### 2.5 决策触发

需要确定安全芯片以满足以下需求：
1. FIDO2 私钥安全存储要求
2. CTAP2 ES256 签名功能
3. ClientPIN 加密通道实现
4. 成本控制在 $20-31 目标内

---

## 三、决策

### 3.1 最终决策

选择 ATECC608A-MAHDA-T 作为 IBio 项目的安全芯片。

**决策详情**：

| 项目 | 决策内容 |
|------|----------|
| **芯片型号** | ATECC608A-MAHDA-T |
| **品牌** | Microchip Technology |
| **安全等级** | EAL4+ |
| **接口** | I2C（高达 1 MHz） |
| **单价** | $1-2 |

### 3.2 决策理由

选择 ATECC608A 的核心理由：

| 理由 | 详细说明 |
|------|----------|
| **价格极低** | 单价仅 $1-2，是市场上最便宜的安全芯片之一，显著降低项目成本 |
| **开源项目验证** | SoloKeys、OpenSK 等知名开源 FIDO2 项目使用 ATECC608A，证明其可行性 |
| **功能满足** | 支持 ES256 签名、ECDH 密钥协商、SHA-256 哈希，满足 FIDO2 全部安全要求 |
| **开发库完善** | CryptoAuthLib 库完善，支持 ESP32 平台，文档丰富 |
| **密钥槽位充足** | 16 个密钥槽位，足够存储多个凭证密钥 |
| **社区活跃** | Microchip 官方支持活跃，社区资源丰富 |

## 七、安全等级评估

### 7.1 EAL4+ 详细说明

**Common Criteria EAL4+ 定义**：
- **EAL4**：方法ically designed, tested and reviewed
- **EAL4+**：EAL4 + Flaw Remediation（缺陷修复）
- **认证机构**：Common Criteria (ISO/IEC 15408)
- **认证时间**：2019 年

**EAL4+ 安全特性**：
| 特性 | 说明 |
|------|------|
| **防软件攻击** | 防止恶意软件提取密钥 |
| **防简单物理攻击** | 防止简单的物理篡改 |
| **密钥加密存储** | 密钥在芯片内部加密 |
| **防侧信道攻击** | 基础的侧信道攻击防护 |
| **真随机数生成** | 内置 TRNG |

**EAL4+ 不满足的特性**：
| 特性 | 说明 |
|------|------|
| **防高级物理攻击** | 不能防止高级物理攻击（如激光攻击） |
| **防电磁攻击** | 基础防护，不能防止高级电磁攻击 |
| **防故障注入攻击** | 基础防护，不能防止高级故障注入 |

### 7.2 EAL4+ vs EAL5+ vs EAL6+对比

| 安全特性 | EAL4+ | EAL5+ | EAL6+ |
|----------|-------|-------|-------|
| **防软件攻击** | 强 | 强 | 强 |
| **防简单物理攻击** | 中 | 强 | 强 |
| **防高级物理攻击** | 弱 | 中 | 强 |
| **防侧信道攻击** | 中 | 强 | 强 |
| **防电磁攻击** | 弱 | 中 | 强 |
| **防故障注入攻击** | 弱 | 中 | 强 |
| **适用场景** | DIY、消费级 | 商业级 | 企业级、金融级 |
| **典型芯片** | ATECC608A | STSAFE-A100 | OPTIGA Trust M, SE050 |

### 7.3 FIDO2 安全要求满足情况

| FIDO2 要求 | ATECC608A 满足情况 | 说明 |
|------------|---------------------|------|
| **私钥不可导出** | 满足 | 密钥在芯片内部生成，永不离芯片 |
| **私钥不可读取** | 满足 | 密钥只能用于签名，不可被读取 |
| **密钥隔离** | 满足 | 16 个密钥槽位，密钥隔离存储 |
| **密钥加密存储** | 满足 | 密钥在芯片内部加密存储 |

## 八、功能评估

### 8.1 ES256 签名功能

| 项目 | 说明 |
|------|------|
| **算法** | ECDSA with SHA-256 |
| **曲线** | NIST P-256（secp256r1） |
| **公钥格式** | 65 字节（uncompressed） |
| **签名格式** | 64 字节（r + s） |
| **签名时间** | ~50-100 ms |
| **FIDO2 要求** | 必须支持 |

**ES256 签名流程**：
```
1. 计算消息哈希（SHA-256）
2. 调用 ATECC608A Sign 命令
3. ATECC608A 使用私钥签名
4. 返回签名结果（r, s）
```

### 8.2 ECDH 密钥协商功能

| 项目 | 说明 |
|------|------|
| **算法** | ECDH with P-256 |
| **用途** | ClientPIN 加密通道、凭证加密 |
| **共享密钥长度** | 32 字节 |
| **ECDH 时间** | ~50-100 ms |
| **FIDO2 要求** | 必须支持（ClientPIN） |

**ECDH 密钥协商流程**：
```
1. 客户端生成临时公钥
2. 调用 ATECC608A ECDH 命令
3. ATECC608A 使用私钥计算共享密钥
4. 返回共享密钥（用于 AES 加密）
```

### 8.3 SHA-256 哈希功能

| 项目 | 说明 |
|------|------|
| **算法** | SHA-256 |
| **输入长度** | 任意长度 |
| **输出长度** | 32 字节 |
| **哈希时间** | ~10-50 ms |
| **FIDO2 要求** | 必须支持 |

**SHA-256 使用场景**：
- 挑战哈希
- rpIdHash 计算
- PIN 哈希（左 16 字节）
- 消息签名哈希

### 8.4 密钥存储功能

| 项目 | 说明 |
|------|------|
| **密钥槽位** | 16 个 |
| **密钥类型** | ECC（P-256, P-384） |
| **密钥生成** | 硬件生成，私钥永不离开芯片 |
| **密钥权限** | 可配置（不可读、可签名、可 ECDH） |
| **密钥加密** | 芯片内部加密存储 |
| **密钥销毁** | Reset 命令覆盖密钥槽位 |

### 8.5 密钥生成功能

| 项目 | 说明 |
|------|------|
| **生成方式** | 硬件 TRNG 生成 |
| **生成时间** | ~50-100 ms |
| **私钥存储** | 自动存储在指定槽位 |
| **公钥输出** | 可读取公钥（65 字节） |
| **FIDO2 要求** | 必须支持 |

**密钥生成流程**：
```
1. 调用 ATECC608A GenKey 命令
2. ATECC608A 使用 TRNG 生成密钥对
3. 私钥存储在指定槽位
4. 返回公钥
```

### 8.6 防篡改检测功能

| 项目 | 说明 |
|------|------|
| **检测类型** | 电压检测、温度检测 |
| **触发条件** | 异常电压、异常温度 |
| **响应动作** | 锁定芯片、销毁密钥（可选） |
| **FIDO2 要求** | 推荐 |

## 九、风险评估

### 9.1 安全等级风险

| 风险 | 描述 | 可能性 | 影响 | 缓解措施 |
|------|------|--------|------|----------|
| **EAL4+ 不足以防止高级攻击** | EAL4+ 不能防止高级物理攻击、电磁攻击、故障注入攻击 | 低 | 高 | 1. DIY 项目使用场景风险低<br>2. 配合物理封装防止简单攻击<br>3. 用户不存储高价值凭证 |

**风险评估结论**：
- EAL4+ 对 DIY 项目足够
- SoloKeys 已通过 FIDO2 认证（L1），证明可行性
- 高安全要求场景不推荐使用 ATECC608A

### 9.2 供应链风险

| 风险 | 描述 | 可能性 | 影响 | 缓解措施 |
|------|------|--------|------|----------|
| **芯片假冒风险** | 购买到假冒芯片，功能异常 | 中 | 高 | 1. 选择官方分销商（DigiKey, Mouser）<br>2. 验证芯片序列号<br>3. 测试芯片功能 |
| **供应链中断风险** | Microchip 供应链中断，无法购买 | 低 | 中 | 1. 提前采购备用芯片<br>2. 关注供应链动态<br>3. 替代方案（OPTIGA Trust M） |

### 9.3 开源库集成风险

| 雎险 | 描述 | 可能性 | 影响 | 缓解措施 |
|------|------|--------|------|----------|
| **CryptoAuthLib 集成复杂度** | CryptoAuthLib 集成到 ESP-IDF 需一定工作量 | 中 | 中 | 1. 参考 SoloKeys/OpenSK 集成方案<br>2. 使用 ESP-IDF I2C 驱动<br>3. 逐步调试验证 |
| **库版本兼容性风险** | CryptoAuthLib 版本更新导致兼容性问题 | 低 | 中 | 1. 使用稳定版本<br>2. 关注库更新动态<br>3. 测试验证更新 |

### 9.4 功能限制风险

| 风险 | 描述 | 可能性 | 影响 | 缓解措施 |
|------|------|--------|------|----------|
| **密钥槽位不足风险** | 16 个槽位可能不足以存储所有凭证 | 低 | 中 | 1. 合理分配槽位<br>2. 使用 resident credential 策略<br>3. 优先使用服务器存储凭证 |
| **无 RSA 支持风险** | ATECC608A 不支持 RSA 签名 | 无影响 | 无 | FIDO2 不要求 RSA，ES256 满足需求 |
| **无 AES 支持风险** | ATECC608A AES 功能有限 | 低 | 低 | 可使用 ESP32-S3 硬件 AES |
